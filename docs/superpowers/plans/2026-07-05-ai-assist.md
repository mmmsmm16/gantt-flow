# アプリ内 AI アシスト実装計画（AI サイクル）

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking. コア（Task 1）と純関数（Task 3 の decisions/preview）は TDD（失敗テスト→実装→緑→コミット）で進めること。API 実装ノートは `.superpowers/sdd/ai-map.md` §8、UI の正は `scratchpad/ai-assist-mock.html`（フロー承認セクション含む）。

**Goal:** コンサルタントがヒアリングメモ（自由テキスト）を貼ると、AI が現プロジェクトへの変更提案（`BatchOp` 列）を生成し、**フロー図の上に「仮ノード/仮エッジ」として直接描画**する。ユーザは**フロー上で 1 個ずつ 承認/否認/修正**（カードリストは補助・非フロー系の判定場所）し、**承認分だけを 1 スナップショット＝undo 一発**で core コマンド経由で一括適用する。手順書ドラフト（`set_procedure`/`add_step`）も同じ承認フローに載る。**既定は完全オフライン**、AI はオプトイン（既定オフ）、API キーはユーザ自身のもの（Anthropic / Azure OpenAI）。

**Architecture:**
- **`runBatch`/`BatchOp`/`BatchOpSchema` を `@gantt-flow/core` へ昇格・決定論化**（`runBatch(p, ops, idGen, now)`）。mcp-server は core を import して追随（write-through 動作は不変・回帰テストで保証）。**共通 zod 検証層（`ProposalsSchema`/`parseProposals`）も core に置く**＝プロバイダ差で品質ゲートが変わらない最終防衛線。
- **プロバイダ抽象は desktop 側**（`ai/provider.ts`）。共通パイプライン: プロンプト構築 → プロバイダ呼び出し（JSON テキスト取得・ストリーミング進捗）→ **core の `parseProposals` で `BatchOp[]` を厳密検証** → 検証済み提案のみ UI へ。Anthropic は公式 `@anthropic-ai/sdk`（`dangerouslyAllowBrowser`・`zodOutputFormat`/`messages.stream`）、Azure OpenAI は生 fetch（`response_format: json_schema`→`json_object` フォールバック）。
- **オフライン既定・オプトイン。`aiEnabled=false` のとき通信コード（fetch/SDK）へ一切到達しない**（実装ガード＋スパイテストで固定）。有効化レバーは 2 段: ①設定トグル（`useUI.aiEnabled`・localStorage `gf-ai`）②CSP（`tauri.conf.json` の `connect-src` に `https://api.anthropic.com` と `https://*.openai.azure.com` **のみ**追加）。
- **API キーはユーザのローカルのみ**。既定はセッション中メモリ保持、「この PC に保存」チェックで `gf-ai-key-*` localStorage に平文保存（UI に明記）。**SettingsFile エクスポート・autosave/backups・.gflow に絶対含めない**（構造的に Project/SettingsFile へキーを入れない＝テストで固定）。
- **承認は UI ローカル状態（undo 非対象）**。`decisions`（op index → 承認/否認/未判定）＋ `edits`（インライン修正）＋ ref-DAG 却下波及の純関数 `resolveApproved`。**プレビューは生成直後に一度だけ `runBatch`（決定論 idGen）→ `reconcileProject` して id を確定**し、decisions 変更時は**再 runBatch せず表示のフィルタ/スタイルだけ**変える。**適用時のみ `resolveApproved` の ops を本番 uuid で `runBatch` し直し `commit` 経由で 1 undo**（reconcile は commit が担当）。
- **提案⇔ノード対応**は preview の `BatchResult.aliases`/`created` を突合するマッピング層（純関数・単体テスト）。フローとカードは**同一 `decisions` 状態を共有**。

**Tech Stack:** TypeScript（core は純 TS・依存追加なし） / React 18 + Zustand 4（desktop） / `@anthropic-ai/sdk`（**desktop に新規追加**） / zod（core 既存・共通検証） / vitest。**Rust/Tauri 側の変更は CSP 2 ドメイン追記のみ**（capability 無関係・cargo 変更ゼロ）。

**設計出典:** `docs/superpowers/specs/2026-07-05-ai-assist-design.md`（確定設計）／ `.superpowers/sdd/ai-map.md`（コード地図＋API ノート・§8）／ `scratchpad/ai-assist-mock.html`（UI の正・フロー承認＝主戦場）。

## Global Constraints

- **`packages/core/src/sync/*`（`reconcileFlow.ts` / `reconcileProject.ts` / `autoPlace.ts` / `tidy.ts` / `bands.ts` 等）は一切触らない。** プレビューは既存の `reconcileProject`/`scenarioFlow` を**呼ぶだけ**。適用の reconcile は既存 `commit` が担当（`runBatch` は reconcile しない設計を維持）。
- **`packages/core` は UI/OS 非依存を維持**。React / Tauri / ブラウザ専用 API を core に持ち込まない。**`batch.ts` の core 移設は可**（zod は core 既存依存）。プロバイダ層（fetch/SDK）は desktop のみ。
- **`aiEnabled=false` で通信コード不到達**（実装ガード＋テストで固定）。**CSP は 2 ドメインのみ**（`https://api.anthropic.com` / `https://*.openai.azure.com`・ワイルドカード全開禁止）。
- **API キーの不混入 3 経路をテストで固定**: ①`SettingsFile`（`collectSettings` 出力）②autosave/backups（localStorage の Project シリアライズ）③`.gflow`（container の `project.json`）。キーは `gf-ai-key-*` localStorage＋メモリのみに存在し、Project にも SettingsFile にも入れない。
- **ID 生成は必ず `idGen`（`IdGen`）注入**。`runBatch` は本番で `uuid`、プレビューで決定論カウンタ、テストで `counter()`＋固定 `now`（`'2026-07-05T00:00:00.000Z'`）を注入しバイト安定にする。core で直接 UUID / `new Date()` を呼ばない。
- **新しい store アクションは必ず `dualwindow.ts` の `ACTION_CLASS` へ 1 行足す**（データ変更＝`'forward'`）。`apps/desktop/test/dualwindow.test.ts` が「全 store アクションが分類済み」を機械 assert する安全網。`decisions`/`edits`/プレビュー等の UI ローカル状態は store アクションでないので対象外。**undo は `commit` 経由で 1 回**（承認分の全 op が undo 一発で戻る）。
- **モデル ID は正確な文字列**: `claude-opus-4-8`（既定）/ `claude-sonnet-5` / `claude-haiku-4-5`。**API 実装は `ai-map.md` §8 の流儀**（公式 SDK・`messages.parse`/`messages.stream`・`zodOutputFormat`・typed errors → UI 文言・`stop_reason:'refusal'` を先に確認）。
- **UI タスク（Task 2 の設定タブ / Task 3 / Task 4）は実装者自身が puppeteer-core（`mcp__claude-in-chrome` でも可）で実画面のスクリーンショット確認を完了条件に含める**。Vite ブラウザモード（`npm run dev -w @gantt-flow/desktop`, http://localhost:5173）で検証。
- **UI セルフチェック（ディスパッチ条件）:** リネーム（onBlur コミット）／ダブルクリック編集／右クリック文脈／キーボード操作／Delete（選択対象のみ・グローバル暴発しない）／Escape で編集キャンセル（`inputBehaviors.ts` の `cancelEditOnEscape`）／選択・スクロール追従／パン等の既存操作を壊さない／**ダーク/ライト両テーマで崩れない**（`styles.css` トークン準拠）。
- **実 API を叩くテストは書かない（すべてモック）。** プロバイダ層はネットワークをモックして「組み立てたリクエストの形」を検証。実 API のスモークは**ユーザ実機で 1 回**（手順は Task 5 に記載）。E2E probe は `MockAiProvider`（設定に依らずテスト用に注入）で行う。
- コミットメッセージは日本語 conventional（`feat:`/`fix:`/`docs:`）。`git add` は**明示パス**のみ。
- **レビュー傾斜:** Task 1 は**コントローラの直接 diff 監査＋mcp write-through 回帰**（migration-safety は永続化変更なしのため不要）。Task 2 / 3 / 4 は専任レビュー（code-review）。最終（Task 5）は**opus によるブランチ全体レビュー**。**フルワークスペーステスト（`npm test --workspaces` / `npm run typecheck --workspaces`）は Task 5 の 1 回のみ**。各タスク内は当該ワークスペースのテスト/型チェックに限る。
- **依存: 1 → 2 →（3 → 4）→ 5。** Task 2 と Task 3 は provider IF（`ProposalRequest`/`AiProvider`/`requestProposals` が `BatchOp[]` を返す）が確定すれば一部並列可だが、`styles.css` / `App.tsx` / `useUI.ts` / `store.ts` を両者が触るためファイル競合に注意（**本計画は 2→3 逐次を既定**とし、並列時は provider IF を先にコミットして土台化する）。

export type DecisionState = 'pending' | 'approved' | 'rejected';
export type DecisionMap = Record<number, DecisionState>;      // op index -> 判定（未登録 = 'pending'）
export interface ProposalEdit { name?: string; assignee?: string } // インライン修正（v1 は名称・担当）
export type EditMap = Record<number, ProposalEdit>;

// ref-DAG 却下波及: producer(ref を持つ op) が rejected/disabled なら、それを参照する下流を無効化（連鎖・fixpoint）。
export function resolveApproved(ops: BatchOp[], decisions: DecisionMap): {
  apply: number[];                       // 適用する op index（approved かつ非 disabled・元順）
  disabled: Map<number, string>;         // op index -> 理由（例: '依存先が否認されたため'）

  edgeOps: Map<number, [Id, Id]>;        // add_dependency op -> [fromTaskId, toTaskId]
  nonFlowOps: number[];                  // フローに置けない op（set_procedure/add_step/add_issue/set_detail/upsert_asset）

  result: BatchResult;                   // runBatch(project, ops, makePreviewIdGen(), PREVIEW_NOW)
  view: FlowLevelView;                   // reconcile+tidy 済みプレビューフロー（対象 level）

**規範的挙動:**
- **プレビュー id 安定（危険地帯 4 の確定設計）:**
  1. 生成完了時、`applyEdits(rawOps, edits)` を畳み込んだ `ops` を固定し、`buildAiPreview(project, ops, level, scope)` を**一度だけ**実行。`buildAiPreview` は `runBatch(project, ops, makePreviewIdGen(), PREVIEW_NOW)` → `ensureLevelView` → `reconcileProject(_, makePreviewIdGen())` → `tidyFlowView` で対象 level の view を得る（`scenarioFlow` と同じ射影経路）。**全 op（却下含む）で作る**ので view は全提案ノードを含む。
  2. `decisions` 変更（承認/否認）では**再 runBatch しない**。フロー/カードは view と `nodeMap` を固定したまま、各 op の decision（＋`resolveApproved` の disabled）で**スタイル/可視性だけ**変える（仮=琥珀破線・承認=緑実線・却下=打消し減光・無効=減光・既存=減光）。
  3. **インライン修正**（名称）は `edits` を更新し、表示ラベルを**その場で上書き**（node id は不変）。担当変更は最終適用の op 引数へ反映し、ノードには担当テキストを更新表示するが、**プレビュー内でのレーン再配置はしない（v1 の割り切り・適用後の reconcile が正しく再配置）**。名称/担当いずれも id を揺らさないため承認状態の対応が壊れない。
  4. **適用**は `resolveApproved(ops, decisions).apply` の index から `ops` を引き、`store.applyApprovedBatch(applyOps)` が **本番 `uuid`** で `runBatch(project, applyOps, uuid, now)` → `commit(result.project, 'AI提案を適用')`。プレビュー id は捨てる（表示専用だった）。commit が reconcile と 1 undo を担当。
- **`resolveApproved`（fixpoint）:** producer = `ref` を持つ op（add_task/upsert_task/upsert_asset）。各 op が「消費する ref」を集める（add_task/upsert_task の `parent`、add_dependency の `from`/`to`、set_detail/set_tobe/add_io/add_issue/set_procedure/add_step の `task`）。消費 ref が producer を指し、その producer が `rejected` or 既に `disabled` なら当該 op を `disabled`（理由付き）に。既存 taskId を指す消費は無効化しない。反復して連鎖を収束（`bands.ts`/`tidy.ts` 同様の visited/反復）。`apply` = `decisions[i]==='approved'` かつ `!disabled.has(i)`。**未判定（pending）は適用しない**。
- **`buildProposalNodeMap`（危険地帯 5）:** `buildAiPreview` は runBatch 前に task 生成 op（add_task/upsert_task）と upsert_asset へ `ref = op.ref ?? '__p'+index` を付与してから実行する（`aliases` が全生成物を index で引ける全単射になる）。`opToTaskId[i]` = task 系は `aliases['__p'+i or op.ref]`、対象系（set_*/add_io/add_issue/add_step）は `task` を `aliases`/既存 id で解決した taskId。`edgeOps[i]` = add_dependency の from/to 解決。`nonFlowOps` = フローに描けない op（手順書/課題/detail/資料）。**単体テスト必須**（producer/consumer 突合・非フロー op の分類・存在ノードへの寄せ）。
- **`applyApprovedBatch`（store・ACTION_CLASS='forward'）:** 実装は `const now = new Date().toISOString(); const { project } = runBatch(get().project, ops, uuid, now); commit(project, 'AI提案を適用');`（`uuid`/`commit` は store 既存）。`dualwindow.ts` の `ACTION_CLASS` に `applyApprovedBatch: 'forward'` を追加（`dualwindow.test.ts` が未分類を検出）。作成系だが FOCUS_INTENT には足さない（承認 UI が対象を持つのでその場リネーム導線は不要）。
- **AI パネル（ドロワー・モック §2/§4/§5 が正）:** `AssetLedger.tsx`（`aside.drawer`）の前例で右ドロワー。状態: `memo`（textarea）／`phase: 'idle'|'streaming'|'ready'|'error'`／`rawOps: BatchOp[]`／`decisions`／`edits`／`preview: AiPreview|null`／`progress: string`。
  - メモ→「提案を生成」=`requestProposals(req, onProgress)`（`onProgress` で `progress`/件数を更新＝ストリーミング進捗）。完了で `rawOps` 確定→`buildAiPreview`→`preview` セット。
  - カードリスト（補助・全 op 一覧）: 各行に kind タグ・要約・[承認/修正/否認]・status バッジ。フローに出る op は「フローで表示 →」。**フローとカードは同一 `decisions`/`edits` を共有**（どちらの操作も即同期）。却下波及（`resolveApproved` の disabled）で下流カードが「無効」表示。

## 付録: 実装者への指示（曖昧さの解消）

- **プレビュー id 安定の一言まとめ:** 生成直後に**一度だけ** `runBatch`（決定論 idGen）＋`reconcileProject` して view と `nodeMap` を固定。以後、承認/否認は**再計算せず表示スタイルだけ**変える。**適用時のみ本番 uuid で `runBatch` し直し `commit`**。名称/担当のインライン修正は id を揺らさない（ラベル上書き・担当は適用後 reconcile で再配置）。
- **共通 zod 検証は core に置く**（`ProposalsSchema`/`parseProposals`）＝プロバイダ差で品質ゲートが変わらない最終防衛線。desktop に zod を追加しないため、Anthropic の `zodOutputFormat` にも core の `ProposalsSchema` を渡す（npm workspaces で zod は単一インスタンス）。
- **`aiEnabled` ガードは `requestProposals` の先頭**（provider 構築・fetch・SDK 生成より前）。テストは `useUI.setAiEnabled(false)` 後に fetch/SDK スパイ 0 回を固定。
- **キーは Project にも SettingsFile にも入れない**（構造的保証）。存在場所は `gf-ai-key-*` localStorage とセッションメモリのみ。「この PC に保存」OFF で localStorage を消す。
- **モデル ID は正確に:** `claude-opus-4-8`（既定）/ `claude-sonnet-5` / `claude-haiku-4-5`。

- **新 store アクションは `ACTION_CLASS` 登録必須**（`applyApprovedBatch: 'forward'`）。`dualwindow.test.ts` が未分類を検出。
