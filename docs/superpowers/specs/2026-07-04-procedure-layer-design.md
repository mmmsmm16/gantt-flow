# 手順書・ノウハウ層と現場向け展開（A→B） — 設計

日付: 2026-07-04 ／ ステータス: 構想確定（視覚モックでユーザ確認済み）
視覚モック: claude.ai Artifact「手順書・ノウハウ層 モック」（`design_reference/procedure-mock.html` に同内容を保存）

## 背景・目的

ツールの将来像を「コンサルの調査・整理ツール」から「現場（顧客の業務実施者）の仕事の
進め方支援」へ広げる。現行の Excel 運用（中工程ごとに 1 ファイル。中に小・詳細工程を
書き下し、各業務へ 1 文 1 アクションの手順書。**目的・条件分岐・参照資料**が最重要）を、
.gflow 単一データソース上の「手順書・ノウハウ層」に置き換える。

提供形態は 2 段階（ユーザ合意済み）:

- **A. ハンドブック出力** — .gflow から自己完結 HTML を書き出して現場へ配布。導入コストゼロで価値検証。
- **B. 現場モード** — 共有フォルダの .gflow を現場が直接開く。手順書のみ編集可・常に最新。
  コンサル撤収後も現場だけで手順書が育つ（ユーザ要望: 改訂は現場でも続けられること）。
- C（Web/SaaS 化）は対象外。

## 確定 UX（モックで確認済み）

- **「手順書」タブを新設**（表・フローに続く同一データの第 3 のビュー）。中工程を開く＝Excel の 1 ファイルに相当。
- **サイドバー＝配下工程の縦フロー**（箱＋矢印）ナビ。スクロールに追従する「いまここ」バッジ、
  クリックでその工程へジャンプ、中工程ごとの手順書作成済み率（例 4/6）を表示。
- **本文に工程オブジェクトは描かない**（フロー表現はサイドバーへ一本化）。本文は工程ごとに
  見出し（名称・担当・工数・「フローで表示」リンク）＋目的 1 文＋ステップ列。
- **ステップ** = アクション文（太字）＋目的 1 文＋詳細本文（必要なだけ長く書ける）＋
  条件ボックス＋参照チップ（資料台帳・帳票）＋画像。
- **条件** = 「〜の場合」＋対処。飛び先の工程への実リンクを張れる（例: 差戻し → 工程 1.2）。
- 手順書が無い工程はプレースホルダ（「＋手順を書く」「✨ヒアリングメモからドラフト生成」
  — 後者は AI サイクルで有効化）。
- 章の入口・出口に「▲ 前の工程 ／ ▼ 次の工程」リンク（中工程をまたぐ導線）。
- **閲覧専用の別デザインは作らない**。A＝この画面から編集 UI を除いた見た目を HTML に焼く。
  B＝同じ画面を現場モードで開く。

## データモデル（AI 可読性を最優先）

方針: **機械が判断に使う部分は構造データ、人の文章は Markdown、全要素に安定 ID**。
将来「手順書を読んで仕事をレビューする／実行する AI」に構造のまま渡せること
（ステップ ID を指した指摘ができること）を設計条件とする。

```ts
// project.manual（新設トップレベルセクション。flow とは独立）
interface Manual {
  procedures: Record<Id, ProcedureDoc>; // key = taskId（末端工程）
  assets: Record<Id, AssetRef>;         // 資料台帳
}
interface ProcedureDoc {
  taskId: Id;
  purpose?: string;        // 目的（1 文）
  steps: ProcedureStep[];
  updatedAt: string;       // 最終改訂日時（ISO）
  revisions: Revision[];   // 軽量改訂履歴（追記式）
}
interface ProcedureStep {
  id: Id;
  action: string;          // アクション（1 文）
  why?: string;            // このステップの目的（1 文）
  bodyMd?: string;         // 詳細本文・ノウハウ（Markdown）
  conds: StepCond[];
  refs: StepRef[];
  images: StepImage[];
}
interface StepCond { id: Id; when: string; thenMd: string; targetTaskId?: Id }
type StepRef =
  | { kind: 'asset'; assetId: Id }             // 資料台帳
  | { kind: 'io'; taskId: Id; ioId: Id }       // 帳票（IoItem）
  | { kind: 'task'; taskId: Id };              // 他工程
interface StepImage { id: Id; file: string /* ZIP 内 assets/… */; caption?: string;
  overlay?: unknown /* 注釈レイヤの席のみ予約。実装しない */ }
interface Revision { at: string; note?: string; by?: string /* 既定 = OS ユーザ名 */ }
interface AssetRef {
  id: Id; name: string; desc?: string;
  locator?: { alias: string; relPath: string } | { url: string }; // 場所エイリアス方式（下記）
}
```

- ID 生成は既存規約どおり `idGen` 注入。
- **reconcile（flow 同期）には一切関与しない**。commands の更新範囲規約は
  「core / details / manual のみ・flow 不可」に拡張する（CLAUDE.md も更新）。
- MCP: `get_procedure` / `upsert_procedure` / ステップ単位の upsert 等を tools と BatchOp
  語彙に追加。AI は手順書を構造のまま読み書きできる。

## 既存フィールドとの関係（二重管理の禁止）

- `TaskDetail.how`（業務内容・一行）は**「一行サマリ」としてそのまま存続**し、手順書タブの
  見出し直下に同一フィールドを表示・編集する（書く場所はデータ上 1 つ。表＝サマリ、
  手順書＝詳細という粒度の役割分担であり、二重管理ではない）。マイグレーション不要。
- `TaskDetail.note` ほか他フィールドは変更しない。

## 粒度と整合規則

- 手順書は**子のいない末端工程にのみ**付く。
- 工程に後から子が生えたら: 手順書は**消さない**。手順書タブにバナー
  「この工程は分割されました」＋「子へ移動」導線を出す。`validate.ts` に warning ルール追加。
- **ダングリングを作らない**: `targetTaskId` / `ioId` / `assetId` の対象消失時は保存を壊さず
  「リンク切れ」表示に落とす（validate ルール＋描画側フォールバック。quarantine と同じ思想）。
- 工程削除時に対応する ProcedureDoc も削除（undo はストア層スナップショットで自然に戻る）。

## 縦フローナビの直列化

- 純関数 `deriveProcedureNav(core, 中工程Id)`（bands / milestoneGuides と同型の導出・保存しない）。
- 配下の末端工程を依存関係＋ order で**決定論的にトポロジカル直列化**。並行する工程は
  一列に畳み、「∥ 並行」バッジで示す（厳密な分岐描画はフロー画面の仕事。ナビには過剰）。
- ゴールデンテストで決定論を担保（counter() 注入）。

## 画像と .gflow の ZIP 束ね化（v2 コンテナ）

- 新形式: **ZIP**（`project.json` ＋ `assets/` 画像）。拡張子は `.gflow` のまま。
  docs/05 で計画されていた ZIP バンドルをこのタイミングで実装する。
- 後方互換: 旧単一 JSON は読み込み可（先頭バイト `PK` / `{` で判別）。マイグレーションは
  メモリ上のみ、明示保存で v2 になる（既存の永続化方針どおり）。
- **fsstore（Rust）と MCP サーバ（Node）の両方が新形式を読み書き**する。アトミック保存・
  助言ロックの仕組みは不変（1 ファイルのまま）。
- 画像はクリップボード貼り付け／ファイル選択で追加。内容ハッシュで重複共有。
  参照されない孤児 asset は保存時に掃除。
- お絵描き（注釈）は実装しない。`overlay` の席だけ予約し、後から矢印・枠・文字の
  注釈レイヤを足してもファイル形式の作り直しが起きないようにする（元画像は不変のまま重ねる）。

## 場所エイリアスとリンク状態（環境差の運用・ユーザ確認済み）

背景: 作る側（コンサル）と使う側（現場）でネットワークが異なり、資料パスは互いに開けないのが常態。

- 台帳の場所 = `{ alias, relPath }`（例: alias「現場共有」＋「規程/与信管理規程.pdf」）または URL。
- **alias → 実フォルダの対応表は各 PC のローカル設定**（.gflow には保存しない）。
- リンク状態は 3 値:
  1. **開ける** — 対応表があり実体も存在
  2. **リンク切れ** — 対応表があるのに実体なし（この時だけ警告）
  3. **未接続** — この環境に対応表がない。グレー表示・エラー扱いしない・検査対象外
- コンサル環境では通常「未接続」。現場（B）で対応表を 1 回設定すれば全リンクが生き、
  真のリンク切れ検査もそこで働く。
- ハンドブック出力（A）は出力時点の対応表で実パスに解決して埋め込む。未接続 alias は
  「場所: 現場共有/規程/…」の表記のまま出力する。

## 現場モード（B・将来サイクル）

- 同じアプリの起動モード。**手順書（steps・画像・台帳・改訂メモ）のみ編集可**。
  工程構造・依存・フロー配置・工数等はロック。
- 保存のたび `updatedAt` 自動更新。改訂メモ（任意 1 行）＋記名（既定 = OS ユーザ名）を
  `revisions` に追記（Excel の改訂履歴シートの代替。完全な差分管理はしない）。
- 共有フォルダ＋既存の助言ロックで多人数運用がそのまま成立する。

## ビルド順序（目安・各サイクルで spec → plan → SDD）

1. **ZIP 束ね化（v2 コンテナ）** — 独立した土台工事。fsstore / MCP / 自動保存 / 更新検知の
   全経路。migration-safety-reviewer のレビュー必須。
2. **手順書タブ** — model（manual）＋ validate ＋編集 UI ＋縦フローナビ＋資料台帳＋画像＋
   場所エイリアス。
3. **ハンドブック出力（A）**。
4. **AI 組み込み** — ヒアリングメモ → 工程提案＋手順書ドラフト（同一の承認フロー）。
5. **現場モード（B）** — 編集スコープ制限＋改訂履歴の現場運用。

順序の根拠: 手順書の器が無い状態で AI を作ると手順書対応の作り直しが発生するため、
器 → AI の順は固定（ユーザ合意）。3 と 4 の入れ替えは可。

## テスト方針

- core: manual スキーマの roundtrip、validate ルール（非末端の手順書・ダングリング参照）、
  `deriveProcedureNav` のゴールデン（決定論 idGen）。
- 永続化: v2 ZIP の save→open ラウンドトリップ、アトミック書き込み、旧 JSON 読み込みの
  後方互換フィクスチャ、マイグレーションはメモリ上のみ（明示保存まで書き戻さない）。
- UI: 実ブラウザ probe（タブ表示・ナビのスクロール追従とジャンプ・画像貼り付け）＋
  スクリーンショット確認。

## 対象外

- お絵描きキャンバス（注釈レイヤは `overlay` の席のみ）
- 手順書の Excel／印刷出力（必要になったら HTML 出力から起こす）
- 版の完全差分・ロールバック（revisions は追記メモまで）
- C（Web/SaaS 化）、アカウント・権限管理（現場モードは起動モードで分ける）
