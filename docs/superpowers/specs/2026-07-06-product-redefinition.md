# gantt-flow 製品再定義（2026-07-06）

製品全体を10ドメイン＋横断＋カテゴリ標準で体系監査（92観点・53の欠落・9の削除候補・14の簡素化）した結果の再定義。
動機は3つ — **日常ワークフローへの最適化 / 機能が多く重い・複雑の解消 / 足りない能力領域の補完**。
本書は「何を残し・何を統合し・何を足すか」の意思決定材料。**削除・統合は提案止まり**（実装は承認後）。

---

## 1. 北極星

gantt-flow の北極星は、業務変革コンサルタントが **「ヒアリング→WBS構造化→フロー同期→手順書→納品」という中核ループを、単一データから迷わず速く一日中回せる既定オフラインの日次ツール**であること。
そして同じ1データから、顧客が理解できる**手順書HTML・スイムレーン業務フロー図・As-Is/To-Be 改善効果（工数・LT・自動化率・課題の軸）を1ファイルで納品しきれる**こと。

肥大した機能は**削らずに段階開示と統合で覆い**、初見のコンサルでも「表を編集すればフロー図と手順書と効果が同期する」核心体験へ最短で到達させ、**案件をまたいで型を再利用**できることで「毎日開く＋説得力ある納品」を同時に満たす。

## 2. 中核ループ

1. **入口** — ヒアリングメモ／Excel下書きの貼り付け／CSV取込で素材を入れる
2. **WBS構造化** — アウトライン表で速記（大/中/小/詳細×担当×前後×工数×I/O×課題）、全項目表で一括点検/横断入力
3. **フロー同期** — `reconcileFlow` が「表1行⇄ノード1個」を保ちつつスイムレーン図へ投影（手動配置・pinned を壊さない・冪等）
4. **手順書** — 末端工程に目的・手順・分岐・画像を肉付け（`project.manual`）
5. **納品** — 手順書HTML・フロー図PNG/SVG・工程表Excel・課題一覧・改善効果レポートを書き出す

土台に **lint 納品前ゲート**（未完成を出さない）、安全網に autosave・世代バックアップ・助言ロック・外部監視。
**滞在時間の最大は(2) WBS編集**で、ここの入力速度が製品全体の体感速度を決める。
現状の弱点は **(5) 改善効果が既定OFF・示唆がMCPに閉じ・案件横断の再利用が無い**等で、ループの出口と回遊が弱いこと。

## 2.5 実務グラウンディング（2026-07-06 本人確認）

§1-2 の中核ループは当初コードからの**推測**だった。本人への確認で以下が判明し、**優先順位を「捕捉速度」中心に組み直す**（本節が §7 ロードマップの読み替え根拠）。

- **領域**: 製造業の現場（設計・製造中心、案件により全般）。語彙は帳票・工程・属人化・手戻り・リードタイム(LT)・停滞・標準化に寄せる。
- **使う場面**: ①**クライアント先でヒアリングしながらライブ入力**、②持ち帰って自席で整形。この2モーメントが主戦場（「提案資料作成専用」でも「全工程で常用」でもない）。→ *人前で会話しながら速く打てること*が土台要件。
- **成果物**: As-Is/To-Beフロー図・改善提案書(課題→方策→効果)・工程表/Excel・手順書の**4種すべて**を同一データから出す。どの出力導線も捨てられない。
- **痛点（本人申告・4つとも該当）**: (1)ヒアリング入力・整理 / (2)見栄え・資料化 / (3)課題と方策の洗い出し / (4)As-Is/To-Be作り分け・同期。

**含意（ロードマップ反映）**:
- **最優先＝ライブ捕捉の摩擦ゼロ化**。会話中に1打鍵で工程を足せること → **表からDSL・貼り付け追加・フィルダウン**を Phase 0/1 の先頭へ繰り上げ。
- **LT列・I/O整合**は製造現場の核心指標（滞留・帳票断絶）と一致 → 推測から**確証**へ格上げ（実装済）。
- **「見栄え・資料化」は役員向けコスト表ではない**（FB既反映）。フロー図/手順書の体裁品質と**納品物ハブ**（一括出力）で応える。
- **As-Is/To-Be同期**は reconcile の本旨。シナリオ間の差分可視化を回遊の出口として強化する。

## 3. 要否マトリクス（keep / simplify / merge / cut）

**★=削除・統合を伴う＝承認後に実装**。それ以外（keep・段階開示simplify）は既存を壊さないため今晩の対象。

| 対象 | 判断 | 根拠（要約） |
|---|---|---|
| 表⇄フロー同期（reconcileFlow・1:1・手動配置/pinned保持・冪等） | keep | 「単一データから同期」という製品の根拠そのもの。不変条件で守る。 |
| WBS編集（アウトライン＋全項目表） | keep | 中核ループ最大の滞在層・単一ソース。ただし両表のセルロジックは重複（内部共通化の余地大）。 |
| 手順書/ハンドブックHTML納品 | keep | ゴール『顧客が理解できる納品』の終端。印刷CSS完成度が高い。 |
| lint 納品前チェックゲート | keep | 『未完成を出さない』要。純関数で自然拡張できる高レバレッジ資産。 |
| As-Is→To-Be 改善効果 | simplify | 中核価値なのに既定OFFで日常から外れている → 既定ON化 or 一回きり導線で接続（無効化は設定に残す）。 |
| アウトライン任意列 既定全ON | simplify | 最初に触る中核が情報過多 → 既定を作業名＋担当＋工数に絞り、他は列メニューでオプトイン（トグルは残す）。 |

| **ComparisonDialog の phase トグル** | **cut-candidate ★** | バー強調しか変えず数値不変。既定delta固定でUI/キーを1つ減らせる。 |

## 5. 簡素化（段階開示・統合で「重い・複雑」に答える）

初見コンサルが晒される**常設ツールバー約21〜24ボタン＋多数のオーバーレイ/モード**が認知負荷の主因。**機能は削らず**、既定を絞って段階開示する。

- **『はじめてモード（シンプル表示）』プリセット** — 中核ループのボタン（新規/開く/保存・戻す/やり直し・詳細・出力・手順書・検索）だけ表示し、別ウィンドウ・集中モード・鍵盤再割当・BPMN制御ノードを折りたたむ。設定でいつでも全機能へ。1フラグで描画分岐を束ねる。（段階開示・削除なし）
- **出力メニューの目的別グルーピング** — 顧客へ渡す/データで渡す/図で渡す（削除なし）
- **アウトライン既定列を作業名＋担当＋工数に絞る**（トグルは残す）
- **設定オプトインを『追加機能を有効にする』1セクションへ集約**（tobe/ai/単キー/ミニマップ）
- **フロー図形追加を『図形を追加▾』1メニューへ**（工程追加は独立維持・移設のみ）
- **読み取り専用4オーバーレイを『点検/レポート』ハブへ統合 ★**（ツールバー4→1・削除を伴うため承認要）

（※ 当初案の「単キー/キー再割当を上級者設定へ段階開示」は **Vimモード維持のFB** を受けて撤回。単キー系はそのまま残す。）
