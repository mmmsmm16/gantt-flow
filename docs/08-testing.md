# 08. Testing Strategy

最大のリスクは**同期エンジン**。純粋関数として、決定論的に徹底テストする。

## 1. 同期エンジンの単体テスト（Vitest・決定論）
- `reconcileFlow(core, details, view, idGen)` は純粋関数 → **テーブル駆動のゴールデンテスト**。
- ID 生成を注入（`idGen`）して出力をバイト安定にする。
- シナリオ例（各々、出力 `FlowLevelView` と `SyncReport` を厳密検証）:
  - 作業追加 → ノードが自動配置される。
  - 作業削除 → 孤立として report に上がる（即削除しない）。
  - リネーム → 位置・レーンが不変。
  - 担当変更 → レーンが変わり位置は保持／再配置提案。
  - 親変更（サブツリー移動）→ 別バンドへ移動。
  - 依存追加（既にユーザーが A→判断→B を作成済み）→ 直接 A→B を張らない。
  - I/O 列を入力 → 帳票オブジェクトが現れ、空に戻すと撤去。再入力で**配置は据え置き**（再 reconcile で増殖しない）。
  - 課題/方策列を入力 → 課題オブジェクトが現れ、対象＝当該タスク。対象ノード消失時はタスクへ寄せる。`visible` は保持。
  - **冪等性**: `reconcile(reconcile(x)) == reconcile(x)`。

## 2. プロパティテスト（fast-check）
- ランダムなコアコマンド列を生成し、常に成り立つ不変条件を検証:
  - 各粒度ビューで「対象タスク 1 件 ⇄ タスクノード 1 個」。
  - `pinned` エッジ・制御ノードは同期で消えない。
  - データだけの編集で、生き残るノードの x/y は変わらない（帳票/課題オブジェクトの配置・`visible` 含む）。
  - 「I/O・課題/方策 列が非空 ⇔ 対応オブジェクト 1 個」。課題の `targetNodeId` は常に実在ノードを指す。
  - ダングリング参照が生まれない。
  - 親範囲バンドは常にツリーと整合（祖先関係を正しく反映）。

## 3. マイグレーションテスト
- 各スキーマ版のフィクスチャ → マイグレーション → 現行形と deep-equal。
- 現行版ファイルのマイグレーションが no-op であること。

## 4. 永続化テスト
- ラウンドトリップ: `save` → `open` が（`updatedAt` を除き）一致。
- アトミック書き込み: 書き込み途中（rename 前）でクラッシュしても**元ファイルが無傷**であること。

## 5. 結合テスト（ヘッドレスストア・Vitest）
- Zustand ストアをコマンド列で駆動（UI なし）し、工程表とフローの整合を検証。
- コマンド＋ reconcile を React 抜きで通す。

## 6. E2E スモーク（Playwright on Tauri webview）
- ハッピーパス 1 本: ファイルを開く → 表で作業追加 → フローにノードが出る → 保存 → 再オープン。
- ロジックの網羅は `core` 単体テストに任せ、E2E は薄く保つ。
