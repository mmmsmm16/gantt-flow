# 07. Open Questions（後で整合する未確定事項）

設計の骨格は確定済み。以下は実装前後に詰める論点。各所では「TBD」として参照している。

## 1. 手順一覧表の列項目 ✅ 確定（2026-06-06）
- 列セットは確定。詳細は `03-view-spec.md` §1-2、型は `02-data-model.md`（`TaskDetail` / `ProcessTask.code`）。
- 必須（同期）: 工程No・作業名・担当・粒度・前/次工程。
- 標準: 業務内容・インプット・アウトプット・使用システム・工数・備考（I/O はフローにもオブジェクトで表示）。
- 任意: 処理件数/ボリューム・**課題**・**方策**・例外対応・自動化区分・データ連携先・関連規程/統制・作業難易度(H/M/L)。
  - 「課題・改善メモ」は **課題列＋方策列の 2 列に分割**。両列はフローの課題オブジェクトと連動（**複数可**）。
  - **帳票様式・保管**は独立列をやめ、各 I/O（主に帳票）の任意属性へ統合。
  - **I/O の単位は工程粒度に連動**（中=帳票/小=情報、`IoItem.kind`）。`帳票 ⊃ 情報` は当面**繋がない（フラット）**。将来 `info.partOfDocId?` で昇格余地あり。
- 複数値の持ち方: フローに出る **I/O・課題は安定ID付きの構造化リスト（複数可）**、その他の複数値（使用システム・データ連携先・関連規程・例外対応）は**自由テキスト＝改行で複数記入**。
- 工数は**時間・0.5h 単位**、工程No は**自動採番＋手動上書き**。
- 残課題: 自動化区分・難易度の選択肢ラベル（手作業/システム自動/一部自動、H/M/L）の最終文言は実装時に微調整。

## 3. フローに出す／出さない項目の最終線引き ✅ おおむね確定（2026-06-06）
- フローに出るオブジェクト: 工程（四角）・**インプット/アウトプット（I/O オブジェクト＝帳票/情報）**・**課題（赤四角・対象へ注釈線・表示切替）**
  ＋ 制御ノード（開始/終了/判断/合流）。形・色・同期は `03-view-spec.md` §2-3、型は `02-data-model.md` §4。
- 表のみ: 工数・使用システム・どうやって・備考ほか。
- 複数 I/O・複数課題に対応（I/O は `IoItem[]`、課題は `IssueItem[]`。各々が安定IDを持ちフローに 0..n 個並ぶ）。
- **粒度ビューごとの配置**: I/O の単位がレベルで変わる（中=帳票/小=情報）ため**ノード自体がレベル別**＝配置(x/y)もレベルごとに独立（タスクノードの byLevel と同パターン）。共有 x/y は不要 ✅。
- 残課題（軽微）:
  - 入力色/出力色・赤の具体的なカラーパレットと帳票形/情報チップの見た目（実装時に確定）。
  - 「使用システム」等を工程ノードのツールチップに**補助表示**するかは任意機能として保留。
  - 複数 I/O・複数課題の**自動配置の並べ方**（縦積み・間隔・衝突回避）の詳細は実装時に詰める。

## 4. 階層のレベルスキップ可否
- 大の直下に小（中を飛ばす）等を許すか。
- 当面は**許さない前提**で実装（親は常に 1 段細かい粒度）。許容する場合の表示／同期ルールは後で定義。

## 5. 依存（流れ）のスコープと粒度跨ぎ
- 依存は「同一粒度・同一親スコープ内」で張る前提（`02`/`04`）。
- 粒度を跨ぐ流れ（例: ある大工程の最後の小工程 → 次の大工程の最初の小工程）の表現方法は要検討。
  当面はスコープ内に閉じ、跨ぎは上位粒度のフローで表現する。

## 6. 同期の方向性の将来拡張
- 現状の逆方向同期は「レーン移動 → 担当更新」の 1 点のみ。
- フロー上での作業追加（図で直接ステップを足す → コアにタスク生成）を将来サポートするかは要検討
  （サポートする場合は明示的な「作業を追加」操作とする）。
