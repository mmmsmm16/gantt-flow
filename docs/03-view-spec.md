# 03. View Spec

2 つのビュー（工程表ビュー／フロービュー）の仕様。両者は同じコアデータを別の見せ方で表示する。

## 1. 工程表ビュー（手順一覧表）

「どんな業務を・誰が・どれくらいの工数で・どうやって」を行で書く表。**時間軸ガントではない。**

### 1-1. 2 つの見せ方（両対応）
1. **階層アウトライン表（WBS 風）**: 大工程の下に中・小・詳細をインデントで入れ子表示。展開／折りたたみ可能。
2. **粒度フィルタ**: 「大／中／小／詳細」を選んで、その粒度の行だけに絞り込んで一覧。

```
[アウトライン表示]                          [粒度=小 で絞り込み]
▼ 大1 受注処理（営業部）                      受注内容を確認   営業  田中  10分 ...
  ▼ 中1 注文書受付                            在庫を引当       倉庫  佐藤   5分 ...
    ・小1 受注内容を確認  営業 田中 10分        出荷指示         倉庫  佐藤   5分 ...
    ・小2 在庫を引当      倉庫 佐藤  5分
  ▼ 中2 出荷準備
    ・小3 出荷指示        倉庫 佐藤  5分
▼ 大2 ...
```

### 1-2. 列（確定）

**■ 必須（同期対象・コア）** — フローと連動
| 列 | 説明 |
|---|---|
| 工程No | 階層番号（`1` / `1-2` / `1-2-3`）。**自動採番＋手動上書き**（`ProcessTask.code`） |
| 作業名 | 業務・作業の名称（フローのノードラベル） |
| 担当 | 担当者／部署（フローのスイムレーン） |
| 粒度 | 大／中／小／詳細 |
| 前工程／次工程 | 流れ（フローの矢印） |

**■ 標準** — 「どんな業務を・どうやって・どれくらいの工数で」
| 列 | 説明 | フロー |
|---|---|---|
| 業務内容（どうやって） | 手順・方法の説明 | 表のみ |
| インプット | 入力（**複数可**。中=帳票/小=情報。帳票は様式番号・保管を任意で） | **I/O オブジェクト 0..n（入力色）** |
| アウトプット | 出力（**複数可**。同上） | **I/O オブジェクト 0..n（出力色）** |
| 使用システム | システム／ツール（複数は改行） | 表のみ |
| 工数 | 所要時間。**分（整数）で保持・表示は分/時間切替**。末端に入力（主に中工程）、親は**子の合計を集計** | 表のみ |
| 備考 | 補足 | 表のみ |

**■ 任意（採用済み）**
| 列 | 説明 | フロー |
|---|---|---|
| 処理件数・ボリューム | 1回/月あたりの件数等 | 表のみ |
| 課題 | コンサル視点の課題（**複数可**。各々 方策・対象を持つ） | **赤四角の課題オブジェクト 0..n** |
| 方策 | 課題への対応策（課題とペア） | 課題オブジェクトに併記 |
| 例外・イレギュラー対応 | 通常フローから外れる対応（複数は改行） | 表のみ |
| 自動化区分 | 手作業／システム自動／一部自動 | 表のみ |
| データ連携先 | 次に渡す部署／システム（複数は改行） | 表のみ |
| 関連規程・統制 | 規程・マニュアル・内部統制ポイント（複数は改行） | 表のみ |
| 作業難易度 | H／M／L | 表のみ |

> **帳票様式・保管**は独立列をやめ、各 I/O（主に帳票）の任意属性に統合。
> **I/O の単位は粒度に連動**: 中工程＝帳票、小工程＝情報（`02-data-model.md` §3 `IoItem.kind`）。当面は帳票⊃情報を繋がない（フラット）。
> 複数値の原則: **フローに出る I/O・課題は安定ID付きの構造化リスト**、その他の複数値は自由テキスト（改行）。
> 「表のみ」列はフローへ出さない。**I/O は I/O オブジェクト（帳票/情報）**、**課題/方策は課題オブジェクト**として
> 表を源泉に投影する（`02-data-model.md` §3・§4・§7、本書 §2-3）。

### 1-3. 編集 UI（グリッド＋行インスペクタ）
列が多く（必須＋標準＋任意で十数列）、I/O・課題は**構造化リスト（0..n・子項目あり）**のため、
**グリッド＋行インスペクタの 2 構成**にする（全列を 1 グリッドに詰めない）。

- **グリッド**: 主要列のみ表示（工程No・作業名・担当・粒度・前/次工程＋工数など、表示列はトグル可）。
  階層アウトライン／粒度フィルタはここに乗る（§1-1）。インライン編集は単純なスカラ列（作業名・担当・工数等）に限定。
- **行インスペクタ**（横 or 下のパネル）: 選択行の**全項目**を編集。とくに構造化セルはここで扱う:
  - **I/O（インプット/アウトプット）**: 帳票/情報を**チップで一覧**し、追加/削除/並べ替え。各チップで 名前・`kind`(帳票/情報)・様式番号/保管 を編集。
  - **課題**: 課題を一覧し、各々 課題文・方策・**対象**（工程 or 特定 I/O）を編集。
  - その他の任意列（使用システム・例外対応・データ連携先・関連規程・備考 等）もここでまとめて編集。
- これにより「列数の爆発」を避けつつ、構造化セルを素直に扱える。

### 1-4. 編集操作（コマンド経由）
- 行の追加／削除／リネーム、担当変更、親の変更（インデント変更＝サブツリー移動）、並び替え（`order`）、前後関係（流れ）設定、
  I/O・課題の追加/削除/編集。**前/次工程は同一スコープ（同一親・同一粒度）の兄弟**を参照する。
- これらは `commands/*` を通じてコアを変更し、`reconcileFlow` を呼んでフローへ反映（詳細 `04-sync-spec.md`）。
- **工数**は末端ノードに入力（主に中工程）。親（大／小に分解された中）の工数は**子の合計を集計表示**（導出・読み取り専用）。
- **粒度はツリーの深さに連動**（レベルスキップ禁止）。**親の担当は親自身の割り当て**（子から導出しない）。

## 2. フロービュー（スイムレーン業務フロー図）

### 2-1. 粒度セレクタ
- 表示する工程粒度（大／中／小／詳細）を切り替える。
- 中・小・詳細では「どのスコープ（どの親工程の中）を見るか」も選べる
  （例: 中工程ビュー = ある大工程の中の中工程フロー）。`FlowLevelView.scopeParentId`。

### 2-2. 2 軸グリッド（独立 2 軸）
**縦＝担当／部署レーン、横＝大>中工程の入れ子バンド。** 両者は独立した軸で、交差してグリッドを作る。

```
                 ┌─ 大工程: 受注処理 ─────────────┬─ 大工程: 出荷処理 ───┐
                 │ ┌中: 注文書受付┐ ┌中: 出荷準備┐ │ ┌中: 梱包出荷────┐ │
 営業 ──────────┤ │(開始)→[受注  │ │           │ │ │              │ │
                 │ │      確認]   │ │           │ │ │              │ │
 倉庫 ──────────┤ │      ↓       │ │[在庫引当] │ │ │[梱包]→[出荷] │ │
                 │ │     [在庫引当]┼→│ →[出荷指示]│ │ │     →(終了)  │ │
                 │ └─────────────┘ └───────────┘ │ └──────────────┘ │
                 └────────────────────────────────┴───────────────────┘
   ↑レーン軸（担当/部署・縦）          ↑バンド軸（大/中工程・横、祖先範囲を背景帯で表示）
```

- **レーンの描画**: 担当レーンは**薄い水平線で全幅を区切る**（ラベルだけでなく帯として分ける）。左端にレーン名（担当/部署）の列を置き、縦区切りで本体と分ける。
- **ノード配置**: 各タスクノードは「担当レーン × 祖先バンド」の交点付近（＝該当レーン帯の中）に置かれる。
  位置は手動で微調整でき、同期で保持される。
- **親範囲バンドは導出**: 表示中の粒度ノードの祖先をたどり、中工程・大工程ごとに背景帯を描く（`sync/bands.ts`）。
  - 小工程ビュー → 中工程・大工程のバンドを入れ子で表示。
  - 中工程ビュー → 大工程のバンドを表示。
  - 大工程ビュー → バンドなし（最上位）。

### 2-3. フローオブジェクトの語彙（形・色・由来）

フロー図に置けるオブジェクトは次の通り。**形と色で種類を識別**する。

| オブジェクト | 形 | 色 | 由来 | 同期 |
|---|---|---|---|---|
| **工程（タスク）** | 四角（長方形） | 既定／担当レーン色 | コア | 表⇄図（ラベル＝作業名） |
| **インプット（I/O）** | `kind` で **帳票形** or **情報チップ** | 入力色（例: 青系）。工程の**入力側に添える**（IN） | 表詳細 `inputs[]`（`IoItem`） | 表→図（表が源泉、配置は手動保持） |
| **アウトプット（I/O）** | `kind` で **帳票形** or **情報チップ** | 出力色（例: 緑系）。工程の**出力側に添える**（OUT） | 表詳細 `outputs[]`（`IoItem`） | 表→図（同上） |
| **課題** | 四角（**直角・角丸にしない**） | **赤** | 表詳細 `issues[]`（`IssueItem`） | 表→図、対象へ細い薄線（矢頭なし）、表示/非表示トグル |
| 開始／終了 (start/end) | 角丸／端点 | — | フロー固有 | フローのみ（レーン非依存） |
| 判断 (decision) | ひし形 | — | フロー固有 | フローのみ（**分岐は図でだけ描く**。条件は出力エッジのラベル・レーン非依存） |
| 合流 (merge) | ひし形／小円 | — | フロー固有 | フローのみ（レーン非依存） |
| コメント（付箋） | 付箋／吹き出し | — | フロー固有 | フローのみ（自由テキスト・**同期しない**・`FlowComment`） |

> **帳票/情報（I/O）は工程に「添える」（接続線は引かない）**。**工程の角に重ねて置く ＝ 入力は左上・出力は右下**（工程と重なってよい）。入力/出力は**位置＋色（青/緑）**で示す（業務フロー図の作法＝図書から流れ矢印を伸ばさない）。
> **例外**: その帳票/情報が**フローの起点（最初の工程がその帳票・情報から始まる）**のときだけ、帳票→最初の工程へ**プロセス矢印を引いてよい**（§2-4）。
> **課題・コメントは逆に「重ねない」** — 他オブジェクト（工程・帳票・矢印・他の課題）と**できるだけ重ならない空きスペース**へ自動配置する（§2-3 末尾・`04` autoPlace）。
> **制御ノード（開始/終了/判断/合流）は担当を持たない** → スイムレーンに束縛されず、バンド内に手動配置する。

- **工程**＝作業 1 件＝ノード 1 個（その粒度において）。
- **I/O（帳票/情報）**: 表の I/O が複数あれば**その数だけオブジェクトが並ぶ**（I/O 1 件＝ノード 1 個）。
  **単位は工程の粒度に連動** — 中工程では帳票（例「注文書」、帳票形）、小工程では情報（例「顧客名」「数量」、情報チップ）。
  「情報が集まって帳票になる」関係はあるが当面データ上は繋がない（フラット）。内容は表を live 表示（図側で本文編集はしない＝表が源泉）。
  様式番号・保管は各 I/O（主に帳票）の付随情報として表示できる。既定では**工程の角に重ねて添える（入力=左上 / 出力=右下）**（接続線は引かない）。同名の帳票でも前工程の出力と次工程の入力は別オブジェクト（必要なら手動の同一帳票リンク・§2-4）。
- **課題**: 表の「課題」「方策」が源泉。**課題が複数あれば複数の赤四角**が出る（課題 1 件＝ノード 1 個）。
  各課題は**対象オブジェクト（工程ノード or 特定の I/O ノード）へ細い薄線（矢頭なし）**で結ぶ（対象は課題ごとに選べる）。
  この線は**矢印ではない＝流れ（プロセス矢印）でもない**ため、依存・経路解決には影響しない。表示/非表示を切り替えられる（§2-6）。
- **配置ポリシー（重なり）**: 付随オブジェクトの自動配置は種類で**非対称**にする。
  - **帳票/情報**: 工程の角（入力=左上 / 出力=右下）に**重ねて**添える（コンパクトに紐づけ、重なりOK）。
  - **課題・コメント**: 他オブジェクト（工程・帳票・矢印・他の課題）と**できるだけ重ならない空きスペース**へ寄せる（読みやすさ優先）。
  - いずれも手動で動かせ、移動後は同期で保持。衝突回避の具体アルゴリズムは `04-sync-spec.md` の `autoPlace`。

### 2-4. 線（コネクタ）の語彙

**矢印（流れ）は工程間にだけ引く。帳票/情報には線を引かず「添える」**のが原則（§2-3）。
フロー図に出る線は次のとおりで、**スタイルで明確に区別**する。

| 線 | 役割 | スタイル（例） | 経路解決への影響 | 由来 |
|---|---|---|---|---|
| **プロセス矢印** | 工程→工程の**流れ** | 実線＋矢頭 | **あり**（依存・経路） | コア依存 `derivedFromDependencyId` / ユーザー描画 `pinned` |
| **課題の線（注釈）** | 課題 ↔ 対象（工程 or I/O） | **細い実線・薄色・矢頭なし** | なし | `FlowIssueNote.targetNodeId` |

- **帳票/情報（I/O）には接続線を引かない**。工程の入力側/出力側に**添えて**、配置＋色で IN/OUT を示す。
- **例外（起点トリガ）**: 帳票/情報が**フローの起点**のときだけ、その帳票/情報 → 最初の工程へ**プロセス矢印**を引いてよい
  （帳票が開始ノードの役割。`pinned` のフローエッジ＝`role:"flow"`、source が帳票ノード。経路解決に参加する）。
- **プロセス矢印**: ユーザーが A→判断→B のように経路を作っている場合、同期は直接 A→B を**勝手に張らない**（`04-sync-spec.md`）。
- **課題の線（注釈・矢頭なし）は「流れ」ではない** → 依存・経路解決には影響しない。
- **同一帳票リンク（任意・手動）**: 「前工程の出力＝次工程の入力」が同じ帳票でも、フラットモデルでは**2 オブジェクトが独立**して出る（§2-3）。
  必要なら**ユーザーが両者を結ぶ連結線を手動で引ける**（任意・装飾的な手動コネクタ＝`pinned` の非フロー線。経路解決に影響しない）。既定では引かない。

### 2-5. フロー上の編集と逆方向同期
- **ノードを別レーンへドラッグ → 表の「担当」を更新**（唯一の「フロー→表」逆方向同期）。
- それ以外（配置・分岐・合流・コメント・課題注釈線・I/O/課題の配置・同一帳票リンク）はフロー固有で、コアへは戻さない。
- I/O・課題オブジェクトの**本文は図で編集しない**（表が源泉）。図でできるのは配置と表示状態のみ。

### 2-6. 課題オブジェクトの表示切替
- **一括トグル**: フロー全体で課題オブジェクト（赤四角＋注釈線）の表示/非表示をまとめて切替（課題レイヤの ON/OFF）。
  - 通常のフロー閲覧時は OFF、課題レビュー時に ON、といった使い分け。
- **個別トグル**: オブジェクトごとに `FlowIssueNote.visible` で表示/非表示（一括トグルと併用）。
- エクスポート時も課題レイヤの ON/OFF を選べる想定（成果物に課題を載せる/載せない）。

## 3. ビュー間の対応（まとめ）

| データ | 工程表ビュー | フロービュー |
|---|---|---|
| 作業（タスク） | 1 行 | 1 ノード（四角・その粒度） |
| 担当 | 担当列 | スイムレーン |
| 階層（親・粒度） | インデント／粒度フィルタ | 粒度セレクタ＋親範囲バンド |
| 流れ（依存） | 前後関係 | 矢印 |
| インプット／アウトプット | I/O 列（中=帳票/小=情報を複数列挙） | I/O オブジェクト 0..n（入力色／出力色、帳票形/情報チップ） |
| 課題／方策 | 課題列・方策列（複数行） | 赤四角の課題オブジェクト 0..n（対象へ注釈線・表示切替） |
